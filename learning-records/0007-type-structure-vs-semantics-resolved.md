# 解决了"类型结构 vs 语义"的遗留问题

[[0006-nested-tuple-arrays-and-type-vs-semantics]] 里留下一个用户明确要求"先记住、自己再消化"的问题：为什么"数组套数组套点"（`[number,number][][]`）这个结构就等于多边形？用户后续主动尝试自己解答，过程和结果值得记录。

## Evidence
- 第一次尝试给出的推理有逻辑漏洞：用户认为"如果不收尾闭合，这堆坐标就是几条延伸的直线，没意义了，所以肯定是多边形"。这个推理站不住脚，因为"不闭合的 `[number,number][][]`"本身**不是没意义的东西**——它正好就是 GeoJSON 里的 `MultiLineString`（多条独立线段，比如被截断的路、等高线），和 `Polygon` 的环数组类型标注完全一样，TS 分不出两者。
- 经过反例点出后，用户在第二次回应里给出了准确表述：`[number,number][][]` 这个类型结构本身不携带"多边形还是多线段"的含义，是它所在 GeoJSON 对象的 `type` 字段（`Polygon` vs `MultiLineString`）赋予了它实际语义，`type` 为 `Polygon` 时约定每个坐标数组要首尾闭合，`MultiLineString` 则不需要。并自己举了 `[[[1,2],[2,3]],[[3,4],[5,6]]]` 作为合法结构的例子。
- 补充说明了一点用户未提及的细节：TS 不会检查"首尾坐标是否真的相等"这类值级别的约定，闭合与否纯粹是数据/规范层面的约定，不是类型系统检查的内容。

## Implications
- "类型只描述形状、不描述语义，语义由外部字段/约定赋予"这个心智模型现在已经通过用户自己的反例验证确认内化，可以放心在后续课程（比如泛型里用 discriminated union 描述 GeoJSON Feature 类型时）默认这个理解已经具备。
- 用户会主动用"举反例验证/反推"的方式检查自己的理解，如果反例本身有逻辑漏洞，直接给一个具体的反例（而不是抽象重复规则）是最有效的纠正方式——这次是 `MultiLineString` 的例子起了作用。
