# 嵌套元组数组类型:理解过程曲折,并牵出一个待消化的开放问题

第三课练习二涉及 `[number, number]` → `[number, number][]` → `[number, number][][]` 三层嵌套(点 → 线 → 多边形环)。这个概念花了好几轮解释才理解,过程值得记录:

## Evidence
- 第一次用"剥洋葱,从右往左剥 `[]`"讲解,用户反馈"感觉有点绕"。
- 第二次换成"文件夹嵌套"类比(点=文件,线=装文件的文件夹,多边形=装文件夹的文件夹),同时补充说明"剥类型(拆解,从右往左)"和"造数据(组装,从左往右/由内往外)"是两个相反方向,之前混着讲导致用户追问"你说的不是从左往右吗"。
- 用户理解了"点→线"(`[number,number][]`)但对"线→多边形"(`[number,number][][]`)为什么就变成多边形仍有疑问——这是一个更深层的问题:**为什么"数组套数组套点"这个结构会等于"多边形"?**
- 关键讲清楚的点:TS 类型只描述"形状"(数组套数组套点),不描述"语义"(这堆点到底是多边形的环、还是互不相连的多条线/MultiLineString)。语义是 GeoJSON 规范+使用这份数据的代码共同约定出来的(比如"每个环首尾闭合、第一个环是外边界、后续环是洞"),不是 TypeScript 类型系统检查或保证的内容。
- 用户回应"看了一个 geojson 数据确实如此",并主动总结了前三课的心智模型(联合类型收窄 / interface 命名对象 / 变量与函数类型标注),整体准确,只有 `void`(课文未显式讲解,用户自己从代码悟出)和 `any`("尽量避免"而非"禁止",且有专门一课会讲 `unknown`)两处小误差。
- **用户明确表示这个"类型结构 vs 语义"的问题还要自己再消化思考,尚未完全内化**,要求先记住这个疑问。

## Implications
- 后续如果用户主动追问"类型结构 vs 实际语义"相关问题(比如 GeoJSON 的 MultiLineString/MultiPolygon,或者别的"同构不同义"场景),可以直接承接这里的讨论,不用从头解释。
- "拆解类型"(从右往左剥 `[]`)和"组装数据"(从内往外套 `[]`)这两个方向容易讲混,以后讲嵌套类型时最好一开始就分开点名,避免重复本次的来回澄清。
- `void` 和 `any` 属于用户自学附加内容/course 之外的引申,后续可以在合适的时机(比如讲 `unknown` 的专门一课)正式补上,不用现在专门补课。
