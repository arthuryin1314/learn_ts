第 8 课

tsconfig 与 Vue + TypeScript 集成

目标:看懂一个 tsconfig.json 里关键字段在管什么;知道 Vue 单文件组件(.vue)里怎么给 props/emits 写类型;知道项目里共享类型该放在哪。

tsconfig.json:TS 编译器的配置文件

之前七课我们都在 Playground 里写代码,Playground 后台其实也是按一份默认 tsconfig 在跑。真实项目里这份配置是显式写出来的,常见关键字段: 

{
  "compilerOptions": {
    "target": "ES2020",        // 编译产物用哪个版本的 JS 语法
    "module": "ESNext",        // 编译产物用哪种模块写法(import/export vs require)
    "strict": true,            // 开启全部严格类型检查
    "jsx": "preserve",         // 给 React/Vue 的 JSX 语法用,纯 Vue SFC 项目通常不需要
    "moduleResolution": "Bundler",
    "skipLibCheck": true       // 跳过第三方库 .d.ts 文件自身的类型检查,加快编译
  },
  "include": ["src/**/*"],     // 哪些文件参与类型检查
  "exclude": ["node_modules"]
}

不需要背全部字段,记住几个最影响你日常开发体验的:

  • strict —— 建议新项目从第一天就开 true。它其实是一组检查的总开关(比如禁止隐式 any、变量必须初始化才能用等)。项目起步时开,比写了很多代码后再开容易得多——后补 strict 往往会一次性炸出几十个报错。
  • include/exclude —— 决定"TS 到底在检查哪些文件"。遇到"明明报错了但 IDE 没提示"或反过来"某个文件不该被检查却被检查了",先看这里。
  • skipLibCheck —— 第三方库自己的类型定义偶尔有小问题,开启这个可以避免因为别人的 .d.ts 报错卡住你,几乎所有项目都会开。
Vue 项目里常见的多文件拆分
用 Vite 创建的 Vue+TS 项目,通常不是一份 tsconfig.json,而是拆成 tsconfig.json(入口,用 references 指向下面两份)、tsconfig.app.json(你写的组件代码)、tsconfig.node.json(vite.config.ts 这类跑在 Node 环境的配置脚本)。拆开的原因是:浏览器代码和 Node 脚本运行环境不同,能用的全局类型(比如 Node 的 process)也不同,分开检查更准确。你不需要自己设计这个结构,脚手架(npm create vue@latest)会帮你生成好,遇到看不懂的字段回来查这一课就行。

Vue 单文件组件里的 TypeScript

在 <script> 标签上加 lang="ts",这个组件里的代码就按 TS 检查: 

<script setup lang="ts">
import { ref } from 'vue'

interface Props {
  title: string
  count?: number
}

const props = defineProps<Props>()

const emit = defineEmits<{
  update: [value: number]
  close: []
}>()

const localCount = ref<number>(0)

function increment() {
  localCount.value++
  emit('update', localCount.value)
}
</script>

几个和你熟悉的"运行时写法"对照一下:

  • defineProps<Props>() —— 用泛型传入一个 interface,比运行时写法 defineProps({ title: String, count: Number }) 精确得多,比如 count?: number 里的 ? 直接表达"可选",运行时写法要额外写 required: false。
  • defineEmits<{...}>() —— 每个事件名对应一个"参数元组"(数组形式),update: [value: number] 意思是触发 update 事件时必须带一个 number 参数;close: [] 表示这个事件不带参数。写 emit('update', 'abc') 传错类型会直接报错。
  • ref<number>(0) —— 大多数情况下 TS 能从初始值自动推断类型(ref(0) 已经能推断出 Ref<number>),显式标注主要用在初始值不能表达完整类型的场景,比如 ref<string | null>(null)。

共享类型放哪:types 目录只是约定,不是强制

组件内部局部用的类型(比如上面的 Props),就近写在 <script setup> 里,不需要拿出去。当一个类型被多个组件、多个模块共享(比如你的"图层配置" LayerConfig,可能同时被地图组件、图层列表组件、API 请求函数用到),才值得拆到单独文件,常见做法是项目根目录下建 src/types/,按业务概念拆文件(比如 src/types/layer.ts),用 export interface/export type 导出,各处 import type { LayerConfig } from '@/types/layer' 引用。

判断要不要拆出去的简单标准:改一下这个类型,要跑去改几个文件的 import?如果只影响一个组件,留在组件里;如果影响三四个地方,拆出去能避免到处复制粘贴同一份字段定义。

为什么 .vue 文件的类型检查工具不是 tsc

tsc(TS 官方编译器)只认识 .ts 文件,不知道怎么解析 .vue 文件里 <template>、<script>、<style> 混在一起的结构。Vue 官方提供了 vue-tsc,专门先把 .vue 文件"翻译"成 TS 能理解的形式再检查,这也是为什么 Vue+TS 项目的 package.json 里做类型检查的命令通常是 vue-tsc --noEmit,而不是直接用 tsc。IDE 里能有类型提示,靠的是 Volar(Vue 官方 IDE 插件)在背后做同样的事情。

小测验

1. 一个已经写了几个月、没开 strict 模式的 TS 项目,现在想把 tsconfig 里的 strict 改成 true,会发生什么?
2. defineProps<Props>()(泛型写法)相比 defineProps({ title: String })(运行时写法),主要优势是什么?
3. 为什么 Vue 项目做类型检查用 vue-tsc 而不是直接用 tsc?

动手练一下

这一课需要一个真实的 Vue 项目环境,Playground 不够用了。如果手头没有现成项目,先起一个: 

npm create vue@latest my-ts-practice
# 勾选 TypeScript,其他选项按默认或按自己喜好
cd my-ts-practice
npm install
npm run dev

项目建好之后:

  1. 打开 tsconfig.json(或 tsconfig.app.json),找到 strict 字段,确认它是 true。
  2. 在 src/components/ 下新建一个 LayerCard.vue,用 <script setup lang="ts">,定义一个 Props interface(至少包含 name: string、visible: boolean、一个可选字段),用 defineProps<Props>() 接收。
  3. 给这个组件加一个按钮,点击时用 defineEmits 定义并触发一个事件(比如 toggle-visible),带一个 boolean 参数。
  4. 在 App.vue 里引入并使用这个组件,故意传一个类型错误的 prop(比如 visible 传字符串),观察 IDE 或终端里 vue-tsc 报错长什么样。

跑通之后把你写的 LayerCard.vue 贴给我看看。
