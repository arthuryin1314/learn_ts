第 5 课

泛型入门

目标:能读懂并写出泛型函数、泛型接口、泛型 class,理解它们比 any 好在哪。

问题:怎么写一个"对任何类型都适用"的函数

假设要写一个"原样返回"的函数,不管传进来什么类型。第一反应可能是用 any: 

function identity(value: any): any {
  return value;
}

const x = identity("hello");
x.toFixed(2); // 不报错!但运行时会崩溃,因为 x 实际是字符串,没有 toFixed 方法

问题在于:用了 any,TS 就彻底放弃了对这个值的类型跟踪——传进去是 string,返回值也应该还是 string,但 TS 完全不知道,所以 x.toFixed(2) 这种"把字符串当数字用"的错误在编译期查不出来,只能等运行时崩。这正是 [[0003-compile-time-vs-runtime-understood]] 里讲过的——一旦标成 any,编译期的保护就没了。

泛型函数:把"类型"也当参数传

泛型解决的就是这个问题——用一个类型占位符(通常叫 T),让 TS 记住"传进去的是什么类型,返回的就是什么类型":

function identity<T>(value: T): T {
  return value;
}

const y = identity("hello"); // TS 自动推断 T 是 string,y 的类型就是 string
y.toFixed(2); // 报错:Property 'toFixed' does not exist on type 'string'.

const z = identity(42);      // 这次 T 被推断成 number
z.toFixed(2);                // 合法

<T> 里的 T 不是关键字,只是个占位符名字(习惯上叫 T,也可以叫别的,比如 TValue)。调用的时候你不用手动写 identity<string>("hello"),TS 会根据实参自动推断出 T 是什么——这叫类型推断(type inference)。

泛型约束:用 extends 限制 T 必须满足什么条件

有时候不能让 T 是任意类型——比如下面这个函数需要用到 .length,就必须约束 T 得有 length 属性: 

interface HasLength {
  length: number;
}

function logLength<T extends HasLength>(value: T): T {
  console.log(value.length);
  return value;
}

logLength("hello");   // 合法,字符串有 length
logLength([1, 2, 3]); // 合法,数组有 length
logLength(42);         // 报错:number 不满足 HasLength 约束,没有 length 属性

<T extends HasLength> 读作"T 可以是任何类型,但必须满足 HasLength 这个形状"——注意这里的 extends 和 class 继承的 extends 不是一回事,在泛型里它表示"约束/必须满足",不是"继承自"。

泛型接口:结合第三课的 GeoJSON Feature

回忆第三课的 PointFeature/LineFeature,如果 geometry 的类型能作为参数传入,一个 interface 就能覆盖所有 Feature:

interface Feature<G> {
  type: "Feature";
  geometry: G;
  properties: Record<string, unknown>;
}

interface PointGeometry {
  type: "Point";
  coordinates: [number, number];
}

const pointFeature: Feature<PointGeometry> = {
  type: "Feature",
  geometry: { type: "Point", coordinates: [116.4, 39.9] },
  properties: { name: "天安门" },
};

pointFeature.geometry.coordinates; // TS 知道这是 [number, number],不是 any

不用泛型的话,要么给每种 geometry 单独写一个 PointFeature/LineFeature interface(第三课的做法,适合"种类固定、不太会变"的场景),要么 geometry 只能标成 any 丢失类型信息。泛型让"Feature 的外壳"复用同一份定义,只在使用时告诉 TS geometry 具体是什么。

泛型 class:结合第四课的 MapLayer

泛型也能用在 class 上,结合第四课的抽象类,写一个"只能装某一种图层"的分组:

class LayerGroup<T extends MapLayer> {
  private layers: T[] = [];

  add(layer: T): void {
    this.layers.push(layer);
  }

  renderAll(): void {
    for (const layer of this.layers) {
      layer.render();
    }
  }
}

const vectorGroup = new LayerGroup<VectorLayer>();
vectorGroup.add(new VectorLayer("roads", true, "/data/roads.geojson"));
vectorGroup.add(new RasterLayer("lake", true, "/data/lake.geojson"));
// 报错!Argument of type 'RasterLayer' is not assignable to parameter of type 'VectorLayer'.

new LayerGroup<VectorLayer>() 相当于告诉 TS:"这个分组的 T 就是 VectorLayer",于是 add 方法的参数类型也跟着变成 VectorLayer——试图塞一个 RasterLayer 进去就会报错。这就是泛型 class 的价值:同一份 LayerGroup 代码,可以配出"只装矢量图层的分组"、"只装栅格图层的分组",而不用为每种图层类型重写一遍分组逻辑。
