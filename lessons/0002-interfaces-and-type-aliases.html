第 2 课

interface / type —— 给对象结构起名字

目标:不再用内联对象类型,而是用 interface/type 给对象结构起名字并复用。

回顾:上节课的痛点

上节课你写的 printUser 是这样的:

function printUser(user: { name: string; age: number }): void {
  console.log(user);
}

如果现在要再写一个 updateUser(user: { name: string; age: number }),你得把这段 { name: string; age: number } 再抄一遍。抄两遍还好,抄十遍呢?而且如果哪天要给 user 加一个字段,你得记得改遍所有出现的地方——这正是"起名字"要解决的问题。

interface:给结构起名字

interface User {
  name: string;
  age: number;
}

function printUser(user: User): void {
  console.log(user.name, user.age);
}

function updateUser(user: User): void {
  // ...
}

现在 User 是一个可以到处复用的类型名。以后要改结构,只改这一处定义就行,所有用到 User 的地方自动同步——这就是"工程化"的一个具体体现:改一处,处处生效。

type:另一种写法

TS 里还有个 type 关键字,效果几乎一样:

type User = {
  name: string;
  age: number;
};
interface 还是 type?
日常写"这个对象长什么样"这种场景,两者几乎等价,团队里统一用一种就行,官方文档也建议优先用 interface。type 的优势会在下面"联合类型"这类场景体现——那是 interface 做不到的。本课先记住:两个都能用来描述对象结构,选一个,别混用。

可选属性:不是所有字段都必填

现实中经常有些字段可能没有值。比如一个用户注册时可能还没填年龄: 

interface User {
  name: string;
  age?: number; // 加个 ? 表示可选
}

printUser({ name: "Ella" }); // 合法,age 不传也行
printUser({ name: "Ella", age: 35 }); // 也合法

注意 ? 加在属性名后面,冒号前面。

只读属性:防止意外修改

结合你熟悉的地图场景:一个坐标点,创建之后通常不应该被随便改动:

interface Point {
  readonly lng: number;
  readonly lat: number;
}

const p: Point = { lng: 116.4, lat: 39.9 };
p.lng = 100; // 编译报错!Cannot assign to 'lng' because it is a read-only property.

这在实际项目里很有用:比如一个图层配置对象初始化后不该被别的代码悄悄改掉,readonly 能让 TS 在编译期就拦住这种误操作,而不是等运行时出现莫名其妙的 bug 再去排查。

动手练一下

在 TypeScript Playground 里试试:

interface Layer {
  id: string;
  name: string;
  visible?: boolean;
}

function toggleLayer(layer: Layer): void {
  console.log(layer.name, layer.visible);
}

toggleLayer({ id: "layer-1", name: "路网图层" });
toggleLayer({ id: "layer-2", name: "建筑图层", visible: true });
