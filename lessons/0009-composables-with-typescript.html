第 9 课

组合式函数(Composables)与 TypeScript

目标:知道什么时候该把组件里的逻辑抽成 composable;会写一个带 TS 类型的 composable;知道 TS 怎么推断 composable 的返回类型。

Composable 是什么

你在综合练习二里写的 App.vue,<script setup> 里有两件事混在一起:渲染逻辑(怎么展示图层列表)和状态逻辑(图层数组怎么维护、怎么更新)。现在两件事加起来不到 15 行,看起来没问题。但如果你以后多了"添加图层"、"删除图层"、"按可见性过滤"等操作,App.vue 就会越来越臃肿,而且如果另一个组件也需要操作同一份图层数据,这些逻辑就要复制粘贴一遍。

Composable 就是解决这个问题的:把"和渲染无关的状态逻辑"抽成一个独立的 TS 函数,哪个组件需要就在哪里调用。 

// src/composables/useLayerList.ts
import { ref } from 'vue'
import type { LayerItem } from '@/types/layer'

export function useLayerList() {
  const layers = ref<LayerItem[]>([
    { name: '图层1', visible: true, opacity: 0.5 },
    { name: '图层2', visible: false, opacity: 0.8 },
    { name: '图层3', visible: true, opacity: 1 },
  ])

  function toggleVisible(index: number, value: boolean) {
    layers.value[index].visible = value
  }

  return { layers, toggleVisible }
}

在组件里用:

<script setup lang="ts">
import { useLayerList } from '@/composables/useLayerList'

const { layers, toggleVisible } = useLayerList()
</script>

App.vue 里原来的 ref、数组初始数据、handleToggleVisible 函数全部消失,换成这一行。模板部分完全不用改,因为 layers 和 toggleVisible 的名字和类型没变。

命名约定:use 前缀

Composable 的函数名固定以 use 开头,比如 useLayerList、useMousePosition、useFetch。这不是语法强制,而是 Vue 社区约定——Vue 内部的 API(比如 ref、computed)也是这个前缀。好处是:一眼看到 use 开头的函数,就知道它内部可能有响应式状态、可能用了 Vue 的组合式 API,和普通工具函数区分开。

TypeScript 怎么处理 composable 的返回类型

大多数情况下 TS 能自动推断 composable 的返回类型,不需要你手动标注。以上面的 useLayerList 为例,TS 会自动推断出它返回: 

{
  layers: Ref<LayerItem[]>,
  toggleVisible: (index: number, value: boolean) => void
}

所以在组件里解构出来的 layers 和 toggleVisible,TS 已经知道它们各自的类型,不需要再单独标注。

什么时候需要手动写返回类型?当 composable 逻辑复杂、推断结果不准确,或者你想明确约束"这个函数只能返回这些东西"时,可以显式标注: 

interface UseLayerListReturn {
  layers: Ref<LayerItem[]>
  toggleVisible: (index: number, value: boolean) => void
}

export function useLayerList(): UseLayerListReturn {
  // ...
}

但对于普通的 composable,依赖 TS 的自动推断就够了,不用每个都手动写返回类型 interface。

Composable 和普通函数的区别
普通工具函数(比如格式化日期的 formatDate())里不会有 ref/computed 这类 Vue 响应式 API。Composable 里有——它的核心价值是"把响应式状态和操作这个状态的方法打包在一起",让组件拿到的不只是一个计算结果,而是一份"活的"响应式数据。这也是为什么命名用 use 前缀专门区分:提醒调用者"这里面有响应式状态,不只是纯计算"。

什么时候该抽成 composable

两个判断标准,满足任意一个就值得抽:

  • 复用:同一段逻辑要在 2 个以上的组件里用。
  • 复杂度:单个组件的 <script setup> 超过 50-80 行,其中有一块相对独立的逻辑(比如"图层管理"、"表单验证"、"数据请求"),可以考虑单独抽出去,让组件本身只剩"把数据和 UI 连起来"这件事。

不要过度抽取:每个 ref 都抽成 composable 是过度设计。只有当"这块逻辑放在这里让人觉得乱"或者"这块逻辑明显可以在别处复用"时再动手。
