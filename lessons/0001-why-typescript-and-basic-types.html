第 1 课

为什么用 TypeScript + 基础类型标注

目标:看懂 TS 相对 JS 到底多了什么,能给变量/函数/数组/对象写出基础类型标注。

TypeScript 到底是什么

一句话:TypeScript = JavaScript + 一层静态类型检查。你现在写的 Vue/JS 代码,原样复制成 .ts 文件基本都能跑——因为 TS 是 JS 的超集,所有合法的 JS 语法都是合法的 TS 语法。

关键的一点,也是最容易被误解的一点:类型标注不会改变代码运行时的行为。TS 编译时会检查类型对不对,但编译完产出的就是普通 JS,类型信息全部被擦除。类型检查只是"写代码时的安全网",不是运行时的功能。

对你的意义
你有 Promise/async-await、数组方法、模块化这些扎实基础,这些在 TS 里完全一样,不需要重学。TS 加的是一层"这个函数期望什么参数""这个对象长什么样"的显式约定——这正是"工程化"的核心:让团队协作、重构、读第三方库(以后是 Cesium、AI SDK)时不用靠猜。

类型推断:大多数时候不用手写类型

新手最常见的误区是给每个变量都手写类型。实际上 TS 会自动推断: 

let count = 5;        // TS 自动推断为 number,不需要写 : number
let name = "Ada";     // 自动推断为 string

只有在 TS 猜不出来的地方(比如函数参数)才需要你手写类型。这是本课接下来的重点。

函数:参数和返回值

TS 猜不出一个函数会被怎么调用,所以参数类型必须手写;返回值类型可以省略(TS 会根据 return 推断),但显式写出来能让意图更清楚,也是团队协作里更推荐的写法。

function add(a: number, b: number): number {
  return a + b;
}

const greet = (name: string): string => `Hello, ${name}`;

数组和对象

let ids: number[] = [1, 2, 3];

function printUser(user: { name: string; age: number }): void {
  console.log(user.name, user.age);
}

下一课会讲怎么用 interface/type 给对象结构起名字,避免每次都写这种内联类型——今天先建立"对象也可以有类型"这个概念。

一个要避开的坑:any

any 会让 TS 完全跳过这个变量的类型检查——相当于把这部分代码打回纯 JS。刚开始写 TS 时,报错太多容易忍不住到处标 any "先跑起来再说",但这会让类型系统形同虚设。遇到不确定的类型,先用 unknown 或者干脆让 TS 自动推断,这个话题后面会有专门一课细讲。

小测验

1. 关于 TypeScript 编译后的代码,以下哪个说法正确?
2. let count = 5; 这行代码中,count 的类型是什么?
3. 给函数参数和返回值加类型,应该写在哪里?
4. 把一个变量标注为 any 会带来什么后果?

动手练一下

打开 TypeScript Playground(在线运行,不用装任何东西),把下面这段代码粘贴进去: 

function add(a: number, b: number): number {
  return a + b;
}

add(1, "2"); // 故意写错,看 TS 怎么报错

观察编辑器里的红色波浪线报错信息,再把 "2" 改成 2 看报错消失。然后自己试着写一个 printUser 函数,参数是一个 { name: string; age: number } 的对象,故意传错一个字段类型,感受一下报错长什么样——这就是 TS 给你的即时反馈循环,以后读报错会越来越快。
